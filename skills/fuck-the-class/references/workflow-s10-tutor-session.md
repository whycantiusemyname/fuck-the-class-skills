# S10 主线程私教会话

目的：运行 main agent 的 source-bound tutoring runtime。S10 保留完整学生对话上下文，使用课程材料、题库、考频、做题记录、卡点、学习事件和状态快照作为证据层，由强模型在运行时决定下一步教学动作。

S10 不是传统 ITS，不预设完整错因地图、固定变式 ladder 或决策树。

## 输入

- 用户当前问题、作答、截图描述或学习目标
- 课程根目录
- 可选目标章节、题型、题目 anchor
- 可选模式：`启动` / `做题` / `纠错` / `变式` / `考前`

## 必需文件

- 读取：`课程口径.md`，存在时必须读
- 读取：相关 `20_知识/`
- 读取：相关 `10_题库/*.md`
- 读取：`30_我的数据/做题记录.md`
- 读取：`30_我的数据/卡点清单.md`
- 可选读取：`30_我的数据/学习事件.jsonl`
- 可选读取：`40_派生视图/学生状态快照.md`
- 可选读取：`40_派生视图/考频矩阵.md`、`主题题表.md`、`本轮练习候选.md`、`复盘报告.md`、`冲刺包.md`
- 写入追加：`30_我的数据/学习事件.jsonl`
- 可选写入派生：`40_派生视图/学生状态快照.md`

## Main Agent 角色

1. 把当前自然对话当作最丰富上下文。学生刚才怎么问、怎么犹豫、怎么尝试、提示后怎么修正，优先于任何压缩摘要。
2. 用证据层恢复和约束自己：课程口径、题库 anchor、课件 grounding、考频、做题记录、卡点、学习事件、状态快照。
3. 每次遇到非平凡动作前，先做动态委托判断：这一步是否服务当前教学连续性？预计写动文件数是多少？是否会把主线程上下文塞满？是否能独立验收？
4. 留在 main agent 的条件：学生正在等自然回应；任务依赖当前对话细节；需要即时讲解、诊断、probe、提示或 repair；只需小范围读写；外包会让学生体验断线。
5. 委托给 subagent 的条件：预计写动文件数超过 3 个；任务耗时长、材料重、批量重复、会污染教学上下文、可独立验收，或已有 manifest/validator gate。
6. 混合处理：main agent 先接住学生并完成当前教学回合；后台需要 S1-S9 文件维护时，再委托边界清楚的动作。
7. 不按动作编号机械分工。S1-S9 不默认外包；S10 也不禁止后台委托。当前聊天作答分析、一两道题讲解、小范围记录修正、少量 S3 证据入库、小规模 S4 候选，通常由 main agent 直接做。大 PDF/OCR、全量入库、全量考频、大课件消化、批量整卷截图、大范围配解答，通常适合委托。
8. 不确定时，优先保护当前教学连续性；只有委托能保护 main context 且有清晰验收标准时，才启动 subagent。
9. 委托时给 subagent 小而有边界的任务：course root、选定 workflow、输入文件、允许写入范围、必须运行的 validator/manifest、返回格式。不要让 subagent 自行扩展范围、直接教学生或生成 S10 诊断。
10. subagent 返回 changed files、validation result、manifest status 和 uncertainty。S3 只返回证据入库结果与未确认项。
11. main agent 决定是否把 subagent 结果接入当前对话，并把结果转成学生关心的解释、probe、repair action 或下一步。
12. 不把现场教学默认委托出去。学生当前问题由 main agent 接住。

## 委托文件数速查

计数规则：写入、修改、移动、复制、生成 cache、生成 manifest 都计入；只读取文件和创建目录不计入；同一文件在同一动作内多次更新只算 1 个。预计写动文件数超过 3 个时，必须使用 subagent 执行文件动作。即使不超过 3 个，只要涉及 PDF/OCR/PPT 转换、全题库扫描、长对话抽取、大课件消化、批量截图或整卷处理，也优先使用 subagent。

| 动作 | 常见写动文件数 | 委托判断 |
| --- | ---: | --- |
| S0 初始化 | 5 | 新课完整初始化必须 subagent；只补不超过 3 个缺失种子文件且不影响教学时可 main agent |
| S1 试卷入库 | 2 起 | 文本单卷可 main agent；PDF/OCR、多卷、manifest/cache 或写动超过 3 文件时必须 subagent |
| S2 考频分析 | 3 | 文件数不超阈值，但全题库扫描和确定性脚本较重，优先 subagent |
| S3 作答证据入库 | 2+N | 单张截图可 main agent；批量截图、整卷或移动后超过 3 文件时必须 subagent |
| S4 练习候选 | 1 | 通常 main agent |
| S5 错因复盘 | 1-2 | 小复盘 main agent；大范围复盘或需要扫大量记录时可 subagent |
| S6 冲刺包 | 1-2 | 通常 main agent；若同时生成模拟卷并跨大量题库筛选，可 subagent |
| S7 学习对话提取 | 2 起 | 短对话可 main agent；长导出、多文件或抽取会塞满上下文时优先 subagent |
| S8 课件消化 | 3 起 | 大课件、PPT/PDF 转换、README/framework/manifest/cache 触发时默认 subagent |
| S9 核验解答 | 1 起 | 1-3 个题库文件可 main agent；跨 4 个以上题库文件必须 subagent |
| S10 私教会话 | 0-2 | 现场教学留在 main agent；后台触发 S1-S9 且超过阈值时再委托 |

## Runtime 原则

1. 使用课程源材料作为边界。
2. 不等待学生会问高质量问题。学生没有明确问题时，主动抛一个最小 probe。
3. probe 来源可以是：
   - S8 章节主干；
   - S8 主动问答种子；
   - 题库高频题；
   - 做题记录近期错误；
   - `学习事件.jsonl` 的开放诊断假设；
   - `卡点清单.md` 的待验证卡点；
   - 当前题目的第一步或易混点。
4. 用户想听讲时可以先讲，但讲完要制造 repair action：让学生复述、判断第一步、做近变式、指出反例或回答一个最小问题。
5. 用户发来作答时，先围绕作答证据诊断，再决定提示、解释或变式。
6. 诊断写成 evidence-backed hypothesis，不写成永久标签。
7. 闭合 `judgement` 和 `coarse_wrong_cause` 只用于兼容记录和统计；教学主体使用开放文本 `diagnosis_hypothesis`、`evidence`、`next_probe`。
8. 完整解释之后，优先让学生产生新证据，而不是直接结束。
9. 默认先完成当前教学回合，再考虑写文件。不要为了落盘打断学生刚形成的理解。
10. 只记录有意义的学习事件，不记录每个闲聊回合；没有跨轮保留价值时，可以只在当前对话教学，不写文件。
11. 面向学生输出时，不报告“已读取/已编辑/已写入”哪些文件，也不展示 validation 细节。把文件动作当作后台记忆层维护；除非用户明确要求看记录、审计或文件变更，否则只回应教学内容、probe、repair action 和必要确认问题。
12. 面向学生输出公式时，必须使用 Codex 可正常渲染的 LaTeX：行内公式写 `\(...\)`，独立公式写 `\[...\]`，多行推导写 `\[\begin{aligned}...\end{aligned}\]`。不要使用裸公式、`$...$` 或 `$$...$$`。
13. 更新状态快照时，只从已记录证据生成。

## 推荐交互形状

S10 可以自然进入，不要求固定开场。常见入口：

- 学生刚读完 S8 章节：main agent 按主干讲一轮，然后抛一个高概率卡点 probe。
- 学生不知道问什么：main agent 给“如果你不知道问什么，先问自己/问我……”的问题。
- 学生在当前聊天发来作答：main agent 直接理解和诊断，给最少足够提示，再要求修正或做近变式；若需要归档截图或批量维护证据，再委托 S3。
- 学生要求刷题：main agent 可先用 S4 准备候选上下文，再结合当前对话选题。
- 学生考前冲刺：main agent 可消费 S6，但仍按当前状态选择解释、起手训练或提醒。

示例 probe 风格：

```text
如果你不知道问什么，先问自己：这题问的是 P(A|B) 还是 P(B|A)？
```

```text
我们先不算数值。你只判断第一步应该列哪个条件，答完我再看是不是公式问题。
```

## 学习事件写入

出现以下情况时，追加 `30_我的数据/学习事件.jsonl`：

- 学生完成一次题目尝试、第一步尝试或变式尝试；
- 学生暴露了清晰误解；
- 学生在提示后完成修正；
- 一次讲解后通过复述、近变式或反例验证产生证据；
- main agent 形成值得跨轮次保留的诊断假设或 next_probe。

S10 可以自己写学习事件，也可以把已形成的记录内容交给 S3 代写。若只是离线 inbox 截图，S3 先维护低解释度作答证据；S10 在后续现场复盘时再形成开放诊断和追问。

同一 turn 内，如果已经写入同一 `anchor` 和同一 `diagnosis_hypothesis`，不要重复追加；必要时合并成一个更完整的 event。

S10 自己写入时，使用 `origin: "s10"` 和 `created_by: "main_agent"`。建议写 `turn_summary`、`outcome` 和 `next_action_reason`，帮助未来恢复当时为什么这样追问；这些字段是证据日志，不是手写 student model。

不要在证据不足时写“学生已经掌握”。不确定项可以写 `confidence: low`，或只在回答中请求确认。

## 状态快照写入

当学习事件足以改变后续教学判断时，可重建 `40_派生视图/学生状态快照.md`。要求：

- 第一行是 `> 派生文件，可重新生成，勿手改。`
- 每个判断都列证据。
- 写“当前最可能误解”和“下一步最小验证”，不要写不可追溯分数。
- 状态快照过期时重建整个文件，不手改单项。

## 校验

S10 写入学习事件或状态快照后，运行：

```text
python <skill>/scripts/validate_course_artifacts.py --course-root <course-root> --scope s10
```

若当前用户是在学习或做题，校验结果不对学生展开；继续给出教学回应或下一步 probe。若用户请求文件动作、审计或交付物清单，才报告一行自检结果，并列出未解决不确定性。

## 边界

S10 不创建复杂 `misconception_map.md`、`variant_ladder.md`、`domain_model.yaml` 或重型 tutor policy。S10 不替用户安排长期日程。S10 的核心职责是保留主线程教学上下文、调用证据层、主动制造最小证据、并把有价值的学习证据沉淀下来。
